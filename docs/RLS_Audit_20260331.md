# Supabase RLS Policy Audit

> **Date**: 2026-03-31
> **Source**: `supabase/migrations/` (001~004)
> **Active Migration**: `001_unified_platform_v2.sql` (View-based bridge strategy, with 001a-001e patches)

---

## Architecture Note

The platform uses a **View bridge strategy** (`001_unified_platform_v2.sql`):
- Real tables: `startups`, `investors` (pre-existing), plus new tables created by migrations
- Views: `pip_startups` (over `startups`), `angel_members` (over `investors`), `pip_gates` (over `gates`), etc.
- RLS is applied to the **real underlying tables**, not to Views (Views inherit RLS from base tables)

## Gap Analysis

### Tables with RLS Enabled but Potentially Incomplete Policies

| Table | Gap | Severity | Note |
|-------|-----|----------|------|
| `angel_card_responses` | No DELETE policy | Low | Responses should not be deleted; upsert pattern used |
| `angel_votes` | No DELETE policy | Low | Same as above |
| `angel_learning_progress` | No UPDATE or DELETE policy | Low | Progress records are append-only |
| `angel_notifications` | No UPDATE or DELETE policy | Low | Notification records are immutable |
| `audit_logs` | No UPDATE or DELETE policy | By design | Audit trail must be immutable |
| `pip_meetings` | No INSERT policy for non-admin | OK | Only admins create meetings |
| `pip_meeting_pitches` | No INSERT policy for non-admin | OK | Only admins assign pitches |
| `mentor_matches` | No INSERT policy for non-admin mentors/teams | Medium | Teams cannot self-book; depends on workflow |
| `mentor_slots` | No INSERT check for team booking | Low | Teams can only read, not insert |
| `in_app_notifications` | No DELETE policy | Low | Users cannot delete notifications |

### Recommendations

1. **High Priority**: Verify `gates` table RLS -- it contains Gate 0/1/2 evaluation scores (sensitive). The `pip_gates` View exposes it; if `gates` lacks RLS, any authenticated user could read all evaluations.

2. **Medium Priority**: Add `service_role` full access policies to new tables (`pip_meetings`, `pip_meeting_pitches`, `angel_card_responses`, etc.) if `ntutec-core` Python scripts need to write to them without going through the API layer.

3. **Low Priority**: The `mentor_matches` table lacks an INSERT policy for non-admin users. If mentors or teams need to create bookings directly (not through admin), a policy is needed.

4. **Note**: The `001_unified_platform.sql` (original, non-v2) creates RLS on `pip_startups` and `angel_members` as if they were tables. The active migration (`001_v2`) correctly applies RLS to the underlying `startups` and `investors` tables instead. Ensure only `001_v2` (+ patches) was executed.
